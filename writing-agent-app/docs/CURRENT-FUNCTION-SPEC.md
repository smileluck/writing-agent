# Writing Agent App 当前功能基线 Spec

更新时间：2026-04-11

## 1. 目的

这份文档用于冻结 `writing-agent-app` 当前版本的产品行为基线。

后续任何 UI、流程、数据结构改动，都应该先对照这份文档：

- 如果只是修 bug，修完后行为应重新回到本文定义的基线
- 如果要改产品行为，应先改本文，再改代码
- 不允许把“最近一次改坏的状态”误当成设计本意

本文区分两类内容：

- `功能基线`：当前版本应该具备的行为
- `已知偏差 / 回归`：代码里当前存在，但不应被视为目标行为的问题

---

## 2. 总体信息架构

顶层导航固定为 5 个入口：

- `项目总览`
- `当前工作区`
- `归档`
- `风格库`
- `设置`

导航职责约束：

- `项目总览` 只负责看项目和默认模型概览，不负责新建文章
- `当前工作区` 负责两件事：
  - 新建文章项目
  - 处理当前打开项目的阶段流转
- `归档` 只看已归档项目
- `风格库` 当前是占位页，不承载主流程
- `设置` 负责工作区目录和模型配置管理

禁止回退到旧设计：

- 不允许恢复整根左侧全高黑色侧栏
- 不允许让 `项目总览` 同时承担“新建文章项目”
- 不允许在编辑模型配置时先展示 provider 预设墙

---

## 3. 模型配置

### 3.1 协议与预设

当前统一协议为：

- `anthropic-compatible`

当前内置 provider 预设：

- Claude 官方
- 阿里云百炼
- 腾讯云
- 百度千帆
- 火山引擎
- 无问芯穹
- MiniMax
- 智谱 GLM
- Kimi Code
- 自定义

### 3.2 配置管理行为

模型配置必须支持：

- 新增配置
- 编辑已有配置
- 明确设置某一项为默认配置
- 同时保存多个 provider 配置

行为约束：

- `新增配置` 才展示 provider 预设列表
- `编辑配置` 必须直接进入编辑表单，不能先回到预设墙
- 编辑已有配置时，`API Key` 可以留空
  - 留空表示继续沿用本机已保存的旧 Key
- 新增第二个及后续配置时，不允许自动把最后一个保存项强行设为默认
- 默认配置必须由以下两种方式产生：
  - 系统首次无配置时，第一项自动成为默认
  - 用户显式点击 `设为默认`

### 3.3 设置页展示规则

设置页应该展示：

- 项目主目录配置
- 全部模型配置卡片

每张模型配置卡片应包含：

- `providerLabel`
- `model`
- `baseUrl`
- `policyNote`（如果有）
- 最近一次测试时间
- `编辑`
- `设为默认`（仅非默认项展示）

设置页不负责：

- 新建文章项目
- 展示所有 provider 预设墙作为默认主视图

---

## 4. 项目总览

### 4.1 页面职责

`项目总览` 页面只负责两类信息：

- 当前默认模型概览
- 活跃项目列表

当前默认模型区域只展示：

- 当前默认配置的 `providerLabel / model`
- `baseUrl`
- `policyNote`（如果有）

不应在总览页展示：

- 所有 provider 配置列表
- provider 预设墙
- 新建文章完整表单

### 4.2 活跃项目列表

活跃项目列表中的每个项目卡片应包含：

- 标题
- 模式
- 当前阶段
- 主题摘要
- 更新时间
- 工作目录

卡片动作：

- `继续`
- `归档`
- `删除`

`归档` 的成功反馈：

- 当前页内直接移除该项目
- 顶部弹出成功提示
- 不自动跳转到 `归档` 页面

---

## 5. 当前工作区

### 5.1 页面职责

`当前工作区` 是创作主入口，必须承载：

- 新建文章
- 当前项目阶段流转

### 5.2 新建文章页

新建文章表单应位于 `当前工作区`，不是 `项目总览`。

表单字段基线如下：

- `你今天想写啥`：必填
- `目标读者`：必填
- `目标字数`
- `工作模式`
- `模型配置`：必填

字段行为：

- 不再手填 `项目标题`
- 项目标题由 `你今天想写啥` 自动生成
- `目标读者` 先根据主题内容给出若干候选项
- 用户可以直接点候选，也可以手动改写
- `目标字数 / 工作模式 / 模型配置` 同一排展示，不能拉成三大块空白

按钮行为：

- 主按钮文案为 `创建项目并开始第一轮`
- 创建成功后直接进入该项目工作区
- 创建成功后自动启动第一阶段 `需求澄清`
- 第一阶段不再需要额外“启动第一轮”按钮

### 5.3 工作区主布局

进入某个项目后，工作区布局顺序必须是：

1. 项目总览头部信息
2. 阶段交接单
3. 主工作区两列

主工作区两列：

- 左列：流程阶段
- 右列：
  - 过程播报
  - 当前阶段成稿
  - 当前阶段的修改 / 选择区
  - 结构化结果区

禁止回退为以下顺序：

- 先修改，再看成稿
- 成稿和过程播报分离得很远
- 中间出现大面积无信息空白板

## 6. 阶段流程

### 6.1 工作模式

快速模式阶段顺序：

- `需求澄清`
- `结构大纲`
- `初稿生成`
- `去 AI 味`

深度模式阶段顺序：

- `需求澄清`
- `立场锁定`
- `调研整理`
- `结构大纲`
- `标题设计`
- `初稿生成`
- `审稿修订`
- `去 AI 味`

### 6.2 阶段文件输出

每个阶段产物都必须自动写入项目目录。

当前后端约定：

- 项目目录名：`YYYY-MM-DD-标题`
- 项目目录内写入 `project.json`
- 各阶段产物按阶段文件名落盘
- 同阶段重写后的新版本按 `v2 / v3 / v4...` 递增

当前阶段产物文件名：

- `需求澄清` -> `01_theme.md`
- `立场锁定` -> `01b_position.md`
- `调研整理` -> `02_research.md`
- `结构大纲` -> `03_outline.md`
- `标题设计` -> `04_titles.md`
- `初稿生成` -> `draft.md`
- `审稿修订` -> `review.md`
- `去 AI 味` -> `final.md`

整篇导出文件：

- `final_export.md`

### 6.3 字数统计

每个阶段产物都必须记录并展示字数。

展示位置至少包括：

- 左侧阶段 rail
- 当前阶段成稿标题区
- 前序材料卡片

---

## 7. 阶段交互规则

### 7.1 生成与交接

普通阶段应支持两类主动作：

- 让当前同事重写这一版
- 确认当前稿，进入下一阶段

这里的“进入下一阶段”必须带清晰语义：

- 文案应体现 `下一阶段名称 + 接手人 + 职能`
- 不能只写人名而不写职能

已废弃行为：

- “接受后还要再点一次生成下一阶段”
- “只推进到下一步但不立刻生成” 这种多余分支

当前冻结要求：

- 用户确认当前稿后，应直接交给下一阶段并开始生成

### 7.2 标题设计阶段

标题设计是强制选择节点。

规则：

- 必须单选 1 个标题
- 未确认标题时，不能进入下一阶段
- 标题确认后，要把确认结果写回阶段产物和后续生成上下文

### 7.3 审稿修订阶段

审稿阶段不能只给一个空文本框。

必须同时支持两种方式：

- 结构化选择修订方向
- 手动输入修改要求

当前基线行为：

- 展示若干可选修订方向
- 支持全选、清空、按选中方向重写
- 用户仍可继续手工输入补充要求

### 7.4 其他阶段的修改建议

非标题、非审稿阶段，也应给出可直接点击的建议项。

要求：

- 建议必须尽量贴当前文章内容
- 不能长期停留在“补细节”“更自然”“更顺”这类机械空话

---

## 8. 过程播报

过程播报是工作区核心模块，必须保留。

### 8.1 作用

过程播报的目标是把 runtime 真实事件翻译成用户能理解的话。

它不是：

- 假步骤轮播
- 自编 1-4 步循环
- 生硬的 SDK 日志直接上墙

### 8.2 展示规则

过程播报应：

- 优先展示解释版自然语言
- 每条允许展开技术记录
- 在首条真实事件到达前，显示“正在准备上下文/加载前序材料”的说明
- 明确告诉用户安静几秒是正常现象，不是卡死

### 8.3 已废弃行为

以下内容不应再出现：

- “你看到的是实时草稿，不是最终定稿”
- 为了凑热闹而伪造的循环步骤

技术记录里如果要显示字数，只保留中性描述：

- `已写约 xx 字`

---

## 9. 成稿区与人工修改

当前阶段成稿区必须：

- 紧跟在过程播报后面
- 明确显示当前阶段文件路径
- 支持直接手动编辑正文
- 支持“恢复到当前版本”
- 支持“保存我的手动修改”

保存手动修改的规则：

- 不能覆盖原版本
- 必须作为新版本继续累积

---

## 10. 终稿与归档

### 10.1 终稿完成

最后阶段 `去 AI 味` 完成后，不再出现“进入下一阶段”。

此时应进入终稿态，提供：

- 继续润色当前终稿
- 打回上一阶段
- 打回初稿生成
- 确认终稿并写入项目目录

### 10.2 归档

归档语义是：

- 从主工作流里收起来
- 不再出现在活跃项目列表
- 只能通过 `归档` 页面进入

归档项目打开后必须是只读态：

- 可以查看
- 可以导出
- 可以恢复
- 不允许继续生成
- 不允许直接继续阶段流转

归档页卡片动作：

- `查看`
- `恢复`
- `删除`

删除应同时删除：

- 项目数据库记录
- 阶段产物记录
- 导出记录
- 项目目录

---

## 11. 风格库

`风格库` 当前是占位页。

当前基线：

- 有单独入口
- 不影响主写作链路
- 不承担创建项目、模型配置、项目流转职责

---

## 12. 已知偏差 / 当前回归

以下问题当前不应被视为设计本意：

### R1. 工作区新建入口被当前活跃项目覆盖

当前代码里，顶部 `当前工作区` 在存在活跃项目时，会优先打开当前项目，而不是始终明确提供“新建文章”入口。

这会导致用户已经选择去工作区新建文章，但表单不出现。

这不是目标行为。

正确基线应该是：

- `当前工作区` 必须始终能明确进入“新建文章”
- 即使已有活跃项目，也不能把新建入口完全吃掉

### R2. 最近改动容易把页面职责重新搅混

尤其要防止以下回退：

- 把新建项目又塞回 `项目总览`
- 把 provider 预设墙重新塞回编辑态
- 把所有模型配置重新堆回总览页

---

## 13. 后续改动约束

后续继续改这个 app 时，先按下面顺序做：

1. 先确认改动是否影响本文某一条基线
2. 如果影响，先改本文
3. 再改代码
4. 改完后按本文逐条回归检查

最低回归检查项：

- 设置页：
  - 新增配置
  - 编辑配置
  - 设为默认
- 项目总览：
  - 只看活跃项目和默认模型
- 当前工作区：
  - 新建项目表单可见
  - 创建后自动启动第一轮
- 标题阶段：
  - 未选标题不能继续
- 审稿阶段：
  - 能选修订方向，也能手写补充
- 归档：
  - 活跃列表移除
  - 归档页可见
  - 归档项目只读
